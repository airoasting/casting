# PLAN.md — 에이전트 팀 빌더 · 라우터 + 3층 스택

> 목적을 입력하면 팀장(오케스트레이터)이 검증된 부품으로 팀을 설계하고,
> 그 팀을 한 번에 복사해 실제 LLM에서 돌릴 수 있게 한다.
> 장기적으로 설계 → 패키징 → 런칭까지 잇는 3층 스택을 지향한다.

이 문서는 이 기능 한정 정본이다. 상위 규칙은 `CLAUDE.md`, 제품 개념은 `PROJECT.md`를 따른다.

## 왜

지금 사이트의 주인공은 "50명 카탈로그"가 아니라 **"목적 → 팀 설계 라우터"**다.
v31의 picker는 목적 5개가 하드코딩된 프리셋 선택이라 "고르기"에 머문다. 사용자는
"목적을 주면 팀원이 선발되고 워크플로우 포함 팀 구조가 설계되는" 경험을 원한다.

## 제품 아키텍처 — 3층 스택

PROJECT.md의 3층 모델(재료 → 제품 → 자산)에 외부 레퍼런스를 한 층씩 매핑한다.
**병합이 아니라 스택이다. 한 번에 한 층씩 짓는다.**

```
목적 입력
  │
  ▼
[1] 라우터·팀 설계   ← 우리 (50 부품 + 하네스 레시피)            = 정적 사이트
  │     검증된 팀 + 워크플로우 + 팀장 지침
  ▼
[2] 패키징·export    ← claude-plugins 포맷(.claude-plugin)        = 사이트가 산출물 뱉기
  │     ①오케스트레이션 프롬프트 ②설치형 플러그인 ③CMA 빌드시트
  ▼
[3] 런칭·운영        ← launch-your-agent / Claude Managed Agent   = 리더의 Claude 스킬
        live 매니지드 에이전트 + 채점 + 스케줄
```

- **옆 선반(Reference, 병합 안 함)**: `affaan-m/ECC`(품질·메모리·보안·채점 패턴),
  `revfactory/harness-100`(어떤 도메인 팀을 만들지 참고 메뉴. 내용 복사 금지, 새로 만든다).
- **경계선(중요)**: 사이트는 [1] 설계 + [2] export(산출물 생성)까지만 한다. 실제 [3] 런칭은
  리더의 Claude Code에서 도는 별도 스킬이 맡는다. 사이트가 CMA를 돌리지 않고, CMA가 먹을
  빌드시트를 만들어 줄 뿐이다(정적 file:// 제약 보존).

### 이번 범위 (v32)

**[1] 전체 + [2]의 export 1종(오케스트레이션 프롬프트 복사)까지.**
[2]의 플러그인·CMA 빌드시트와 [3] 런칭 스킬은 export 훅만 설계에 남기고 **로드맵**으로 분리한다.

## 무엇 — 데이터 모델 (두 층)

| 층 | 이름 | 정의 | 출처 |
|---|---|---|---|
| 부품 | `PARTS` | 단일 역할 = 시스템 프롬프트 1개. v31의 50개 그대로 재사용 | `assets/prompts.js` + A 배열 |
| 레시피 | `HARNESS` | 목적 + 부품 조합 + 팀장 지침 + handoff 순서 + 검토자 | 신설(우리가 작성) |

- 하네스 1개 추가 = 새 프롬프트가 아니라 **조합 한 줄 + 팀장 지침**. 그래서 100까지 확장이 싸다.
- 100은 목표 숫자가 아니라 로드맵이다. v32는 시드 12개로 시작하고, 화면엔 "검증된 팀 N개,
  계속 추가"처럼 **사실대로** 표기한다(CLAUDE.md: 지어내지 않는다 / 안티소울: 규모 자랑 금지).

### HARNESS 스키마(안)

```js
{ id, name,            // 목적 라벨 (예: "리서치 보고서 완성")
  cat,                 // 카테고리 idx
  desc,                // 한 줄 설명
  lead,                // 팀장 지휘 지침(이 팀을 어떻게 엮는지)
  steps:[              // 단계 = 부품 + 입출력
    { part, io }       // part=부품 id, io="입력 → 출력"
  ],
  parallel:[[id,id]],  // 같은 단계 병렬 묶음(선택)
  review,              // 검토자 부품 id (고정 단계)
  mods }               // 이 하네스에서 켤 수 있는 토글 키들
```

## 어떻게 — 화면 흐름

배치: 히어로 바로 아래 기존 `#picker`를 **「팀 빌더」 블록**으로 격상. 50명 카탈로그(`#catalog`)는
그 아래 부품 창고로 둔다.

### STEP 1 · 목적 입력 + 추천

- 큰 입력창에 자유 입력(예: "경쟁사 3곳 분석해 이사회 보고서"). 라우터가 키워드를 하네스의
  이름·설명·부품에 매칭해 **적합 하네스 카드 1~3개를 점수순 추천**.
- 아래 카테고리 4종 칩(리서치·인텔리전스 / 재무·숫자 / 전략·기획 / 문서·커뮤니케이션)으로
  막막할 때 브라우즈. = 자유 입력의 자유도 + 가이드의 안전망.
- 추천 카드 선택 → STEP 2. (매칭 0건이면 가장 가까운 카테고리 제안.)

### STEP 2 · 구조형 팀 + 인라인 토글

별도 페이지 없이 결과 안에서 구체화한다. 토글을 건드리면 팀이 눈앞에서 다시 조립된다.

```
            ┌──────────────────────────┐
            │  LEAD · 팀장(오케스트레이터)  │   고정 · 50명 밖 전용 노드
            │  "<목적> · <옵션 요약>"      │
            └────────────┬─────────────┘
            1 → 2(∥ 병렬) → 3 → 4 …          선발된 단계 노드
            └────────────┬─────────────┘
            ┌────────────▼─────────────┐
            │  REVIEW · 검토자            │   하네스별 고정 1명(검토·품질 섹션)
            └──────────────────────────┘
```

- 각 노드: 역할 배지(LEAD/단계 번호/REVIEW) + 직책 + `입력 → 출력`. 노드 클릭 → 기존 프롬프트 모달.
- 색: 단계는 4색 순환, LEAD=핑크·REVIEW=옐로 고정으로 위계 표시.
- 인라인 토글 4종:

| 토글 | 효과 |
|---|---|
| 사실검증 필요 | research 단계 뒤 2 팩트체커(+7 출처검증) 삽입 |
| 분석 강화(심층) | 하네스별 분석 역할 추가(8/4/11 중) |
| 반박 검토 | REVIEW 앞 49 비판적검토자 삽입 |
| 영문/번역 | 마지막에 27 번역 삽입 |

### export (v32 = 복사 1종)

**⧉ 이 팀 프롬프트 복사** — 클립보드에 오케스트레이션 메타프롬프트 적재:
`목적 + 옵션 + 팀장 지휘 지침(LEAD) + 각 부품 시스템 프롬프트(prompts.js) + handoff 순서 + 산출 형식`.
team 객체 하나에서 생성하므로, 이후 같은 객체로 ②플러그인 매니페스트 ③CMA 빌드시트를
뽑는 함수만 추가하면 [2]가 열린다(export 훅 설계).

6. 동기화: v32 → `output/.../index.html` → 루트 `index.html`. router.js를 양쪽 assets/에 복사.

## 검증 (합격선 9.5, CLAUDE.md §7)

- 12개 하네스 전부 선택 시 최소 팀(≥3 + 검토자) 생성, 깨짐 없음.
- 자유 입력 시 키워드 매칭이 합리적 하네스를 점수순 추천(0건 폴백 동작).
- 토글 4종 on/off가 팀을 눈에 보이게 재조립.
- 복사 버튼이 실제 LLM에 붙여 돌아가는 완결 프롬프트를 만든다.
- file:// 더블클릭에서 프롬프트 전멸 없음(동기 로드 확인).
- 화면 카피는 사실대로(검증된 팀 N개). 규모 과장·미존재 수치 없음.
