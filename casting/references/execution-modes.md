# 실행 모드 · 에이전트 팀 / 서브에이전트 / 폴백

설계한 팀(LEAD + 팀원 + 검토자)을 실제로 돌리는 방법. 핵심은 **각 팀원을 독립 에이전트로** 두는 것이다. 그래야 검토자가 자기 산출물을 채점하지 않고(게이트가 진짜), 독립 단계를 병렬로 돌릴 수 있다.

## 모드 선택 (위에서부터 가능한 것)

| 모드 | 조건 | 방법 | 특징 |
|---|---|---|---|
| 에이전트 팀 | `TeamCreate`·`SendMessage`·`TaskCreate` 사용 가능 | 팀원을 한 팀으로 만들고 공유 작업목록으로 자체 조율 | 발견 공유·상충 토론·실시간 피드백 |
| 서브에이전트 | `Agent` 도구만 가능 | 각 팀원을 서브에이전트로 호출, 파이프라인+팬아웃 | 안정적·어디서나 작동(기본값) |
| 순차 역할극 | 에이전트 도구 없음 | 팀장이 역할을 순서대로 직접 수행 | 폴백 |

대부분의 환경에서 **서브에이전트가 안정적 기본값**이다. 팀 프리미티브가 있으면 그쪽이 더 좋다.

## 실제 도구 호출 (선언이 아니라 호출로)

이 스킬의 가치는 "한 모델의 역할극"이 아니라 **각 팀원이 진짜 별도 에이전트**라는 데서 나온다. 그래서 팀장은 각 팀원을 상상으로 "연기"하지 말고 **실제로 도구를 호출**한다. 팀장이 팀원 산출물을 자기 손으로 이어 쓰기 시작하면, 검토자 독립성이 무너지고 이 스킬을 위반한 것이다.

- **서브에이전트 모드(기본)**: 팀원 1명 = `Agent` 도구 1회 호출. `subagent_type`은 범용(`general-purpose` 또는 `claude`), `prompt`에는 아래 "팀원 에이전트 프롬프트 템플릿"을 채워 넣는다. **병렬 단계는 한 메시지에서 `Agent`를 여러 번 호출**한다. 검토자도 **반드시 별도 `Agent` 호출 1회**(작성한 에이전트와 다른 호출)로 띄운다.
- **에이전트 팀 모드**: `TeamCreate`로 팀을 만들고 `TaskCreate`/`SendMessage`로 단계를 배분·조율한다.
- **폴백(순차 역할극)**: `Agent`·팀 도구가 **전혀 없을 때만**. 이때도 검토는 가능하면 새 턴에서 비평가로 하고, 최종 점수 줄에 `(자기검토 · 독립 아님)`을 병기한다(독립성이 없다는 사실을 사용자에게 숨기지 않는다).

## 역할 프롬프트를 어디서 얻나 (id 구간 추출)

팀원에게 넘길 역할 지침의 **원천은 항상 `agent-prompts.md`의 해당 id 구간**이다(워크스페이스 파일 배치는 그 사본일 뿐, 없어도 여기서 직접 읽는다). id N의 구간은 **`## [N]`으로 시작하는 줄부터 다음 `---`(또는 다음 `## [`) 직전까지**다. 전체 파일(2600줄)을 읽지 말고 그 구간만 뽑아 쓴다. 예: `sed -n '/^## \[N\]/,/^---/p' references/agent-prompts.md` 로 id N 구간만 읽는다(N을 실제 숫자로).

## 병렬 vs 순차 판단

- **순차(파이프라인)**: 한 단계가 앞 단계의 산출물을 입력으로 받아야 하면 순서대로. (예: 요약 → 보고서 작성 → 교정)
- **병렬(팬아웃)**: 서로의 결과가 필요 없는 단계는 동시에 띄우고, 뒤에서 합친다. (예: 마켓 리서처 ∥ 경쟁 분석가 → 둘 결과를 보고서 작성가가 통합)
- 판단 기준: "이 단계가 앞 단계의 산출물 없이 시작할 수 있는가?" 그렇다면 병렬 후보.
- 검토자는 항상 **마지막**, 모든 산출물이 모인 뒤 독립으로.

## 팀원 에이전트 프롬프트 템플릿

각 팀원 에이전트를 띄울 때, 아래 골자로 프롬프트를 만든다.

```
당신은 "<직책> (<English>)" 역할입니다. 아래 역할 지침을 그대로 따르세요.

<references/agent-prompts.md 의 해당 id 시스템 프롬프트 전문>

## 이번 작업
미션: <사용자 목적 한 줄>
당신의 입력: <앞 단계 산출물 또는 사용자가 준 자료>
당신의 산출물: <이 단계가 내야 할 것, 예: "검증된 사실이 달린 조사 메모">

사실·수치가 필요하면 WebSearch/WebFetch로 실제로 찾아 출처를 다세요.
확인 안 되는 값은 지어내지 말고 [확인 필요]로 표시하세요.
자기평가 줄은 출력하지 마세요(품질 채점은 검토자가 따로 합니다).
산출물만 깔끔하게 반환하세요.
```

병렬 단계는 같은 턴에 여러 에이전트를 함께 띄우고, 파이프라인 단계는 앞 에이전트의 반환값을 다음 프롬프트의 "당신의 입력"에 넣는다.

**입력 전달 주의**: 서브에이전트는 부모(팀장)의 대화 맥락을 못 본다. 그래서 그 단계에 필요한 것은 프롬프트 본문에 직접 넣어야 한다. 판단 기준은 **분량**이다.

- 앞 산출물이 작업 대상이고 분량이 감당되면(예: 한 장짜리 문서를 교정·검토) **전문을 통째로** 넣는다. 요약하면 디테일이 사라져 다음 에이전트가 헤맨다.
- 입력이 아주 크면(긴 문서·대용량 자료) 전부를 모든 에이전트에 넘기지 않는다. 컨텍스트가 폭발한다. 첫 단계에서 **구조화 요약/발췌**를 만들고, 이후 단계엔 그 요약 + 해당 단계에 필요한 원문 구간만 넘긴다. 원문 위치(섹션·행)를 표시해 검증 가능하게 둔다.

## 독립 검토자 프롬프트 템플릿

검토자는 **산출물을 만들지 않은 별도 에이전트**다. 그래서 떨어뜨릴 수 있다.

```
당신은 "<검토자 직책>" 역할의 독립 검토자입니다. 이 산출물을 당신이 만들지 않았습니다.
<references/agent-prompts.md 의 검토자 id 시스템 프롬프트 전문>

## 검토 대상
사용자의 원래 목적: <목적>
사용자가 준 자료: <자료 요약 또는 "없음">
산출물: <최종 산출물 전문>

## 할 일
1. 먼저 구체적 결함을 찾으세요(칭찬 금지):
   - 목적이 요구한 걸 빠뜨렸는가
   - 자료·사실과 어긋나거나 지어낸 수치·출처가 있는가
   - 핵심 자리에 빈 [확인 필요]가 있는가, 다음 행동으로 안 이어지는가
   - 구조·분량·톤이 결과물 종류에 안 맞는가
2. 5차원 0~10 채점: 정확성·근거 / 목적·완결성 / 구조·형식 / 실행력 / 언어·톤
3. 구체적 결함이 하나라도 있으면 9.5 미만을 주고, 어느 단계가 고쳐야 하는지 지목하세요.

반환: 결함 목록 + 차원별 점수 + 평균 + (불합격이면) 보강 지시.
```

## 보강 루프

검토자가 9.5 미만을 주면, **결함을 만든 단계의 에이전트에게** 검토자의 구체 지적을 붙여 한 번 더 띄운다. 그 부분만 고쳐 받은 뒤, 검토자가 다시 보고 통과시킨다. 한 번이면 충분하다(무한 반복 금지).

## 팀장(오케스트레이터)의 역할

팀장은 직접 일하지 않는다. (1) 모드 선택, (2) 팀원 에이전트 띄우고 handoff 연결, (3) 산출물 수집·제시, (4) 검토자 띄우고 보강 루프 1회, (5) 최종 결과 + 검토 점수 줄 출력. 토큰을 아끼려면 각 에이전트에 그 단계에 필요한 입력만 건넨다(전체 맥락을 매번 다 넘기지 않는다).
